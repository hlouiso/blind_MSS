 blind_MSS — Optimization log
 Baseline: commit 875e176 (July 2026).  Reference machine: Intel i5-9300H,
 8 threads, N=4 (M=218, tau=65) unless stated otherwise.

--------------------------------------------------------------------------------
0. Pre-baseline optimizations (before this log, from the git history)
--------------------------------------------------------------------------------
  0.a  ZKBoo -> KKW migration                                [commit 30202b6]
       The original prover was ZKBoo-style (N=3, 219 rounds); moving to the
       KKW preprocessing/cut-and-choose model made N a parameter and cut the
       round count to tau = ceil(128/log2 N)+1 — the foundational proof-size
       decision of the project.
  0.b  OpenMP parallelisation                       [commits 537d389, 3a3c261]
       Prove pass 1/pass 2 and the verifier (offline preprocessing check with
       per-thread aux buffers, then the online rounds) run on all cores.
  0.c  Strict minimum M                             [commits 25937bf, c040084]
       Exact KKW soundness formula in params.py instead of conservative
       over-provisioning: smallest M per (N, tau) — less pass-1 work and a
       smaller offline section.
  0.d  Proof-content trimming                       [commits 2e606e9, decf48f]
       Per-party broadcast pairs replacing a redundant global broadcast copy,
       and removal of the hidden party's yp_e duplicate from the proof.
  0.e  Offline-verify fast path (~2-3x)                      [commit 3378c32]
       aux recomputed from the Beaver tapes alone, skipping the circuit
       re-run per opened instance.  Superseded by 8 (masked values), where
       aux depends on the mask flow and the circuit is re-run with zero
       publics; see "mask-flow aggregation" under Ideas below.
  0.f  Library + in-memory pipeline                          [commit d34ce7c]
       File-passing CLI binaries replaced by libblindmss.a and an in-memory
       test suite: no serialisation to disk between protocol phases.
  (ZKBoo-era optimizations before 0.a — e.g. 3397f6f "reduced proof size",
   Oct 2025 — are deliberately omitted: the KKW migration superseded them.)

Baseline numbers at 875e176 (before this log's changes)
  e2e harness (keygen+sign+prove+verify+negative tests) : 22.4 s
  prove ~3.5 s | verify ~2 s | proof size 108.8 MB
  (Note: until the harness fix below, the e2e test ran prove and each verify
   TWICE via a double-evaluating CHECK macro; all e2e times in this log share
   that bias, so their ratios remain valid.  The per-phase bench_pipeline
   numbers are unaffected and authoritative.)

--------------------------------------------------------------------------------
1. Remove redundant per-party H_com from the proof            [commit 075d776]
--------------------------------------------------------------------------------
  What : the per-round h[N][32] field (H_com(seed||x||yp) per party) was
         written to the proof and checked, but never bound into h* before the
         challenge.  The verifier recomputes every input to it from proof
         data, so the check added no soundness: binding is already complete
         via h_j (seeds+aux), h' (all broadcasts, hence x-shares) and h_out
         (all yp shares) — exactly the KKW binding set.
  Also : error-path memory-leak fixes in kkw_prove; kkw_verify M x 32 tables
         moved from stack to heap; proof magic bumped KKW1 -> KKW2.
  Gain : -N*32 B/round (tiny); strictly less data revealed; correctness.

--------------------------------------------------------------------------------
2. Vectorised MPC gates (portable 128-bit vector types)       [commit be4850e]
--------------------------------------------------------------------------------
  What : mpc_ADD/mpc_ADD_verify reformulated branch-free and packed 4 parties
         per GCC/Clang vector_size(16) lane (SSE on x86, NEON on arm64, no
         per-ISA intrinsics).  Verify side pads N-1 slots with inert zero
         lanes.
  Safety: transcript proven bit-identical to the reference implementation
         (random-input equivalence harness, N=4 and N=8, all hidden parties).
  Gain : circuit eval ~2x (building_views 79 -> 49 ms/instance);
         e2e 22.4 -> 14.5 s (with -O3, see 3).
  Note : SUPERSEDED by 8 (masked values, commit 20cf642).  The masked-values
         rewrite reformulated mpc_ADD/mpc_ADD_verify around one broadcast word
         per party and dropped the vector_size(16) packing; the party loops are
         now scalar and left to -O3 -march=native auto-vectorisation (see 3).
         No net regression: 8 is faster overall despite losing explicit SIMD.

--------------------------------------------------------------------------------
3. Build flags                                                [commit be4850e]
--------------------------------------------------------------------------------
  What : default OPT -O2 -> -O3; -march=native on Linux only (Apple clang
         already targets NEON+crypto by default).
  Gain : ~8% alone; enabler for 2.

--------------------------------------------------------------------------------
4. Single-shot AES-CTR tape expansion                         [commit 729721d]
--------------------------------------------------------------------------------
  What : the 64-byte chunk loop paid EVP_EncryptUpdate call overhead ~28k
         times per tape; replaced by one in-place update (same keystream).
  Gain : expansion 8.6 -> 3.7 ms/instance; e2e 14.5 -> 12.8 s.

--------------------------------------------------------------------------------
5. Per-thread scratch reuse in kkw_prove                      [commit 918987f]
--------------------------------------------------------------------------------
  What : each pass-1 instance malloc/freed ~13 MB (tapes, x-shares, aux,
         da_db_all) — ~2.8 GB of allocator churn per proof.  Buffers are now
         allocated once per OpenMP thread and reused across both passes.
  Gain : e2e 12.8 -> 12.45 s; removes per-instance OOM paths.

--------------------------------------------------------------------------------
6. Word-parallel mpc_ADD (no per-bit carry loop on prove)     [commit c4ade15]
--------------------------------------------------------------------------------
  What : two observations kill the 31-iteration bit loop:
         (a) the global carry word equals that of the REAL addition of the
             reconstructed operands: cg = (xg+yg) ^ xg ^ yg;
         (b) each party's carry word is the prefix-XOR of its whole-word AND
             outputs (log2 shift trick, 5 shift/XOR pairs).
         Verify keeps a 31-step scalar recurrence to rebuild DA/DB (it lacks
         the hidden party) but the per-slot work is word-parallel.
  Safety: bit-identical transcript (same equivalence harness as 2).
  Gain : building_views 47 -> 21 ms/instance; e2e 12.4 -> 7.8 s.
         Prove ~1.8 s, verify ~0.9 s (bench_pipeline medians).

--------------------------------------------------------------------------------
Attempted and reverted
--------------------------------------------------------------------------------
  * compute_aux_from_seeds streaming restructure: no measurable gain — the
    old 12-stream gather was already prefetcher-friendly; cost is AES
    expansion (fixed by 4).  Reverted per the "keep only if it improves" rule.
  * GGM / one-tree BAVC (eprint 2024/490 §3): analysed, not implemented.
    It compresses seed openings, which total ~11 KB of a 108.8 MB proof
    (<0.01%): our proof is dominated by the online transcript (aux+msgs,
    12 B/gate x 152504 gates x tau rounds), unlike Picnic/FAEST signatures
    where seeds dominate.  Also our cut-and-choose hides tau/M ~ 30% of
    leaves (dense regime, paths barely merge).

--------------------------------------------------------------------------------
7. Grinding / proof-of-work on the Fiat-Shamir challenge      [commit a79a24e]
--------------------------------------------------------------------------------
  What : FAESTER-style (eprint 2024/490 §4): the challenge hash must end in
         GRIND_W zero bits, found by incrementing a counter ctr (~2^W short
         hashes, one-time, prover side; ctr goes into the proof).  Any forger
         pays the same 2^W per attempt, so the cut-and-choose target relaxes
         to 2^-(128-W): total attack cost stays 2^128 (per RO query:
         2^-W * 2^-(128-W) = 2^-128).  tau = ceil((128-W)/log2 N)+1, M from
         params.py (tables for W in {0,16,24}; build with make W=<W>).
  Gain : N=4, W=16 (default): tau 65->57, M 218->189.
         Proof 108.8 -> 94.7 MB, prove 1.77 -> 1.61 s, PoW ~15 ms.
         Proof format KKW3 (header carries W; ctr after h*).

--------------------------------------------------------------------------------
8. Masked-values online phase (original KKW formulation)      [commit 20cf642]
--------------------------------------------------------------------------------
  What : every wire carries a PUBLIC masked value (value XOR mask) plus the
         parties' XOR-shares of the mask.  Linear gates are public; each
         nonlinear gate broadcasts ONE word per party (shares of the masked
         output) instead of the two Beaver openings (da, db).  Per gate the
         proof carries 8 B (s_e + aux; aux is absent in the ~1/N of rounds
         where party 0 is the hidden one — its com_{j,0} binds it) instead
         of 12; h' hashes half the
         broadcast volume; tapes shrink to 2 words/gate; the masked witness
         d = w XOR lambda_w travels once per round (all mask shares are
         seed-derived, no more witness-offset party).  aux depends on the
         mask flow, so the offline check re-runs the circuit with zero
         publics (masks are witness/public-independent).
  Safety: full suite at N=4 and N=8, ASan+UBSan clean; gate count unchanged;
         round-trip test checks prover/verifier masked-output agreement.
         Proof format KKW4.
  Gain : proof 94.7 -> 57.5 MB; prove 1.61 -> 1.38 s; e2e 7.8 -> 5.5 s.

--------------------------------------------------------------------------------
9. Commitment randomiser r_j restored (KKW Fig. 2)           [commit 73f91b5]
--------------------------------------------------------------------------------
  What : h'_j = H(d || s_all) was a deterministic commitment to the witness
         for opened instances (seed*_j is published, so d and s_all are
         determined by w) — an offline guess-and-check oracle on w, i.e. the
         NIZK was not zero-knowledge.  KKW §3.1 drop r_j deliberately and say
         so, justified for signatures where w is the high-entropy secret key.
         A blind signature needs blindness, whose distinguisher is the SIGNER,
         who already knows sigma, the leaf index and d; only the HM opening
         retains entropy.  Restoring h'_j = H(d || s_all || r_j), with r_j
         revealed only for online rounds, gives back a real ZK simulator.
         Online straight-line extraction is unaffected (d travels in the
         clear; seed extraction goes through the com_hidden RO queries).
  Note : r_j MUST NOT be seed-derived (seed*_j is published for opened
         instances — that would reproduce the bug).
  Cost : +tau*32 B (~1.8 KB).  Proof format KKW4 -> KKW5.

--------------------------------------------------------------------------------
10. Size-neutral cleanup: drop recomputable hashes from the wire  (KKW6)
--------------------------------------------------------------------------------
  What : (a) online h'_j no longer travels — the verifier recomputes it from
         d, s_slots, msgs_e and r_j, and folds the recomputation into the h*
         check (the old memcmp was subsumed by it).  -tau*32 B, paying for 9.
         (b) offline h_out_j no longer travels — yp holds the output-wire
         mask shares, seed-derived like aux, so the verifier recomputes it
         from seed*_j in the same zero-publics run that recomputes aux.
         -32*(M-tau) B.  Offline record: 96 -> 64 B.
         (c) the verifier now requires EOF right after the online section —
         canonical proof encoding, no trailing-byte malleability.
         Also: make SEC=256 selects the rho=256 (post-quantum) parameter
         tables; the default rho=128 tables are a classical bound (README).
  Gain : net -32*(M-tau) B (~4.2 KB at N=4) vs. KKW4; guard tests added
         (h_out witness-independence, prover/verifier h'_j agreement, r_j
         byte-flip rejection).

--------------------------------------------------------------------------------
11. BLAKE3 tweakable-hash migration (binius64 PR #1620)          (KKW7)
--------------------------------------------------------------------------------
  What : every in-circuit SHA-256 replaced by Th built on the raw BLAKE3
         compression function, per BLAKE3_MIGRATION.txt: 336 nonlinear slots
         per compression vs 728 (free message-schedule permutation, XOR
         feedforward), no SHA padding, tweaks in the chaining value for
         free, new 0x03 leaf separator (required: leaf/tree-node domains
         collide under zero-padding otherwise).  Native xmss.c/commitment.c
         follow the same layouts (blake3.c reference, validated against
         official BLAKE3 vectors via the root-flags path); dead SHA-256 MPC
         gadgets removed.  The KKW layer (h_j, h', h*, FS) stays SHA-256.
         Gate model from the plan was exact: predicted 73,096, measured
         73,096.  BREAKS blind-longfellow byte compatibility (README).
  Safety: test_blake3 (official vectors + Th structure + mpc gadget vs
         native for all hidden parties), full suite at N=4/N=8/SEC=256,
         ASan+UBSan clean e2e.  Proof format KKW6 -> KKW7.
  Gain : ySize 152504 -> 73096 (-52%).  Measured (bench_pipeline, N=4):
         proof 57.5 -> 28.8 MB, prove 1.38 -> 0.47 s, verify 0.90 -> 0.26 s.

Audit notes (July 2026, after the masked-values switch)
--------------------------------------------------------------------------------
  * Binding reviewed: h* = H(H(h_j)||H(h')||H(h_out)) commits, before the
    challenge, to seeds+aux (h_j, incl. aux inside com_{j,0}), the masked
    witness d and every party's broadcast stream (h'), and the output-mask
    shares (h_out).  The verifier's grinding check is mandatory and present.
  * New e2e tamper test: single-byte flips at targeted offsets (nonce, h*,
    ctr, offline section, com_hidden, r_j) and random online offsets must all
    be rejected — passes.  While writing it we found and fixed a test-harness
    bug: the CHECK macro evaluated its condition twice, so the e2e test had
    been running kkw_prove twice (two proofs appended to the file) and each
    kkw_verify twice.  Protocol unaffected; e2e timings were inflated ~2x.
  * Two items from this audit were later found insufficient and fixed: the
    deterministic h'_j broke ZK (see 9), and trailing proof bytes were
    accepted (fixed with 9).  h_out_j was never witness-dependent (yp holds
    seed-derived output-mask shares) and no longer travels (see 10).
  * Known accepted limitations: raw-struct serialisation (host endianness);
    mpc_sha256 returns silently on OOM (and hash helpers zero their output
    on EVP/alloc failure instead of aborting).

--------------------------------------------------------------------------------
Ideas not (yet) implemented
--------------------------------------------------------------------------------
  * Mask-flow aggregation for offline verify: recompute aux (and now h_out)
    on the XOR-aggregate of the N tapes (one synthetic party) instead of an
    N-party circuit run — would make the offline check N-independent again
    (matters at N>=16, where it dominates verify).  Correctness argument
    validated (audit 2026-07-09): aux/h_out only consume XOR-aggregates, and
    the mask flow is F2-linear everywhere it is distributed (XOR trivially;
    prefix_xor_shift distributes over XOR; mask_from_bit replicates one bit).
    Needs a dedicated single-party mask-flow evaluator (the compiled
    N_PARTIES loops don't shrink by zero-padding) plus a bit-exactness test
    against compute_aux_from_seeds for N in {4,8,16}.
  * GGM/one-tree seed trees (eprint 2024/490 §3): compresses seed openings,
    which total ~11 KB of a ~58 MB proof here (<0.02%) — not worth it for
    this circuit size; the same paper's grinding idea (7) was the useful one.
